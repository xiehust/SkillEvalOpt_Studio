# 任务集编辑支持 AI 扩充任务

## Goal

让用户在 Studio 编辑已有任务集时直接发起 AI 任务生成，并将审核后的生成结果追加到指定 split，减少离开编辑流程后手工复制任务的成本，同时保持现有“生成结果不自动落库、保存前可人工修改”的安全边界。

## Background

- 任务集详情页会在进入编辑时加载全量任务，并通过 `PUT /api/tasksets/{id}` 整体校验、原子替换。
- 现有 `GenerateTaskSetForm` 已支持单 Skill/Plugin、执行后端、模型、数量、生成指引和超时，并创建异步 `taskgen` 作业。
- taskgen 作业结果已是标准 `TaskItem[]`，当前只能“导入为新任务集”。
- 生成脚本当前只读取 Skill 内容和用户指引，不读取已有任务，生成 ID 默认采用 `task_001` 风格，因此直接追加可能产生内容重复或 ID 冲突。

## Requirements

1. 在非只读任务集详情页的右上方按钮区提供“AI 扩充”快捷入口；点击后加载全量任务、进入编辑模式并直接展开 AI 扩充表单。编辑卡片内仍保留原有扩充入口。
2. 用户可选择扩充目标 split；single 模式固定为 `tasks`，split 模式可选择现有 split，并允许选择尚不存在的可选 `test` split。
3. 复用现有 AI 生成参数和异步 taskgen 作业，不另建同步模型调用路径。
4. taskgen 作业在编辑页内异步运行并轮询状态，期间不得阻塞或重置用户现有编辑草稿；界面提供作业详情链接。
5. 作业结果页明确显示扩充目标，并提供返回原任务集的追加入口；原有“导入为新任务集”能力保持可用。
6. 生成成功后，将结果追加到当前目标 split 的编辑草稿而非直接写入任务集；用户可以继续修改、删除，并通过现有保存动作统一校验后落库。
7. 用户刷新或离开编辑页后，仍可从作业详情页将成功结果恢复追加到原任务集的编辑流程。
8. 合并时不得静默覆盖已有任务：生成脚本先将已有 ID 作为保留 ID 校验，冲突时通过现有二次尝试机制让 AI 重写；前端合并仍遇到冲突时，为生成任务自动分配下一个可用的 `task_NNN` ID，并保留其他字段。
9. 样例任务集保持只读，不提供直接扩充入口。
10. 中英文界面文案保持完整，新增交互具备稳定的测试选择器。
11. AI 生成必须读取目标任务集全部 split 的已有任务摘要，目标 split 优先排列，并明确要求覆盖新的场景、避免跨 split 语义重复；上下文需设置确定性的数量/字符上限，不能让大型任务集无限扩大提示词。

## Acceptance Criteria

- [x] 非样例任务集详情页右上方按钮区显示“AI 扩充”快捷入口；点击后加载全量任务并直接打开扩充表单，样例任务集不显示该入口。
- [x] 用户可从已有可编辑任务集发起 taskgen 作业，并配置 Skill/Plugin、后端、模型、数量、指引和超时。
- [x] single 与 split 任务集均能记录正确的目标 split 和原任务集 ID。
- [x] taskgen 作业在编辑页内显示 queued/running/terminal 状态并提供详情链接；运行期间用户编辑草稿不被重置。
- [x] taskgen 成功后自动追加到目标 split 的当前编辑草稿；此时后端数据尚未改变。
- [x] 用户刷新或离开后，可从作业详情页恢复追加到原任务集编辑流程。
- [x] 用户保存后，原任务与新增任务均存在，任务集元数据计数正确；取消编辑不会写入新增任务。
- [x] 生成脚本检测到已有 ID 冲突时进入现有重试流程；前端合并的最终冲突通过可用 `task_NNN` ID 自动消解，不覆盖或丢失任务。
- [x] 生成提示词按“目标 split 优先、其余 split 随后”的顺序包含受限长度上下文，并明确要求新增覆盖、避免跨 split 语义重复。
- [x] 普通“AI 自动生成新任务集”流程和现有 taskgen 后端行为保持兼容。
- [x] 相关后端/脚本单元测试通过，前端 TypeScript 构建通过。

## Out of Scope

- 自动判断生成任务应如何分配到多个 split。
- 生成完成后绕过人工审核并自动保存。
- 修改任务集持久化格式或引入新的数据库。
